HTTP API

This page documents every HTTP endpoint the device serves. Examples use this device's address (window.location.host). For protected endpoints, run the login command first to create cookies.txt.

Authentication uses a session cookie. POST /api/login with a JSON body {"password":"..."} sets a session cookie valid for 12 hours. Reuse the cookie on later requests. When auth_enabled is false, every endpoint is open and no cookie is required; secrets stay redacted in responses regardless.

Each endpoint carries a badge: Public needs no session, Auth requires a valid session cookie when authentication is enabled.

Create the cookie once:

curl -c cookies.txt -X POST http://HOST/api/login -d '{"password":"YOUR_PASSWORD"}'

Then pass -b cookies.txt on every protected request shown below.

Control API

A generic control surface for reading and changing device settings from single HTTP requests. Six control endpoints plus an image-refresh action, all backed by a live registry. Useful for any HTTP client, for example macro keypads, API Ninja, or Home Assistant.

This page does not hand-list every controllable item. Call GET /api/control/list for the full live catalog of items the device exposes.

Stateless authentication. Besides the session cookie, every auth-gated endpoint on the device (not only Control API) also accepts the admin password in the request header X-Auth-Password: YOUR_PASSWORD. This lets clients that cannot perform the cookie login flow authenticate on every request. It is rate-limited: it shares the login lockout, so repeated wrong passwords trigger a temporary lockout. Send it as a header, not a URL query, so it is not written to the device logs.
Header authentication example
curl -H 'X-Auth-Password: YOUR_PASSWORD' http://HOST/api/control/list
Item object shape. Every read and write returns a uniform item object: name (the config field name), type (bool, enum, or int), value, label (a human-readable string: an enum gets its name, a bool gets On or Off, an int gets the value), and min, max, step. An unknown name, an operation that does not match the item type, or a missing parameter returns HTTP 400.

POST/api/nav/pinAuth

Pin the displayed page. While pinned, all automatic page changes are suspended (auto-rotate, session auto-follow, idle, and default re-home) and the current page is held until cleared. Manual navigation still works while pinned, and the chosen page persists. The pin is in-memory only and resets on reboot. It does not change the auto-rotate setting, it only suspends it while active.

Query parameters: on = 1 to pin, 0 to clear (required). Optional id = a page id from /api/pages to navigate to that page and pin it in one call.
Request (pin current page)
curl -H 'X-Auth-Password: YOUR_PASSWORD' -X POST 'http://HOST/api/nav/pin?on=1'
Request (go to a page by id and pin)
curl -H 'X-Auth-Password: YOUR_PASSWORD' -X POST 'http://HOST/api/nav/pin?on=1&id=7'
Request (clear, resume automation)
curl -H 'X-Auth-Password: YOUR_PASSWORD' -X POST 'http://HOST/api/nav/pin?on=0'
Response
{"pinned":true}

The same state is exposed as the nav_pinned Control item: GET /api/control/get?name=nav_pinned returns {"name":"nav_pinned","type":"bool","value":false,"label":"Off",...}. Toggle or set it like any bool (for example toggle?name=nav_pinned), so a macro key can show Pinned or Auto (On or Off) and flip it. The pin overrides auto-rotate while active.

Behavior notes. cycle wraps from max to min; set and adjust clamp to the range; toggle works on bool items only. The special item name page controls the current page and is the one item that navigates rather than saving config: cycle moves to the next available page, and set selects a page by id. Item names match the config field names used elsewhere in this documentation.
Worked example: cycle the image source and read the label
curl -H 'X-Auth-Password: YOUR_PASSWORD' -X POST http://HOST/api/control/cycle?name=image_display_source

Each call advances the source and returns the new item object. The response label (for example "Solar") can be shown on a key face so the keypad always reflects the current state.
